https://docs.zama.ai/tfhe-rs[Zama TFHE^] wrapper enabling homomorphic encryption in Java using the Foreign Function & Memory API (Project Panama).

== Overview

TFHE Java is a Java wrapper for the https://github.com/zama-ai/tfhe-rs[Zama TFHE-rs library], providing access to Torus Fully Homomorphic Encryption (TFHE) operations directly from Java applications.
This library enables developers to perform computations on encrypted data without decrypting it first, ensuring privacy and security in cloud computing scenarios.

== Key Features

* **Arbitrary Bit-Width Support**: Supports homomorphic booleans (`FheBool`) and arbitrary bit-width signed and unsigned integers (`FheInt` / `FheUint` from 2-bit to 2048-bit, including standard sizes like 8, 16, 32, 64, 128, 256, 512, 1024, and 2048).
* **Flexible Generic APIs**:
** **Type Casting**: Generic `.castInto(Class)` API to cast between ciphertext types (e.g. `fheBool.castInto(FheUint8.class)`).
** **List Extraction**: Extract typed elements from compact or compressed ciphertext lists via generic `.get(index, Class)` (e.g. `list.get(index, FheInt32.class)`).
* **Native Thread Safety**: Thread-safe execution context management via `ServerKey.runWith()` and `ServerKey.callWith()`.

=== Requirements

* **Java 25+** or later (requires Foreign Function & Memory API support)
* **JVM Arguments**: `--enable-native-access=ALL-UNNAMED`
* **Native Libraries**: Automatically included in the JAR (supports macOS ARM64, Linux x86_64, Windows x86_64)

== Building from Source

For most users, the library is available through Maven Central and doesn't require building from source.

However, if you need to build locally

1. First, generate libs/bindings for TFHE: `./mvnw initialize -Plocal`
2. Then, generate jar: `./mvnw package` (or `install` in local Maven repository)

The `local` profile performs the following steps in `initialize` phase:

1. **TFHE-rs download**: Downloads the tfhe-rs source from GitHub
2. **Rust compilation**: Compiles the TFHE Rust library with C API
3. **Binding generation**: Uses jextract to generate Java bindings from C headers

=== Performance Tips

* **Key Reuse**: Generate keys once and reuse them across operations.
* **Memory Efficiency**: Select the smallest appropriate bit-width data types (e.g., `FheUint2` / `FheUint4` / `FheUint8` instead of large default widths) to optimize performance and reduce memory usage.
* **Batch Operations**: Group multiple operations when possible to reduce overhead.
* **Resource Management**: Always use try-with-resources for Arena management.

== Benchmarks

JMH micro-benchmarks (run on Windows 11, OpenJDK 25) show that Project Panama FFM downcall latency is extremely low (~9 ns).

Additionally, the overhead of Java object allocation and GC/Cleaner tracking is negligible (<1%) compared to the CPU time required for FHE computations (~43 ms for homomorphic addition), confirming that developers can safely use immutable API methods without a performance penalty.

[cols="3,1,2,1,5", options="header"]
|===
| Benchmark | Mode | Score | Unit | Description
| `benchmarkTrivialFfmCall` | avgt | ~0.009 | us/op | Baseline FFM downcall latency (9 ns)
| `benchmarkAdditionInPlace` | avgt | ~43255.885 | us/op | Homomorphic addition mutating object in place (~43.2 ms)
| `benchmarkAdditionWithAllocation` | avgt | ~42826.146 | us/op | Homomorphic addition with fresh Java object creation (~42.8 ms)
| `benchmarkSingleThreadedSum` | avgt | ~973721.144 | us/op | Sequential sum of 10 ciphertexts (~973 ms)
| `benchmarkMultiThreadedSum` | avgt | ~977733.033 | us/op | Sum inside Rayon thread context without parallel decomposition (~977 ms)
|===
